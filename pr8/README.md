
```bash

```

Відповідь:
Так як ми досліджуємо qsort(), потрібно розуміти, що це бібліотечна реалізація QuickSort, що потребує лише компаратора, це достатьно швидкий метод сортування даних, проте потрібно розуміти як він працює, ключовим фактором є вибір pivot, тобто елемент умовно посередині відносно якого буде відбуватись сортування, тож в даному завданні було реалізовано компаратор для порівняння цілочисельних даних, а саме int, будемо виконувати дослідження для великого розміру масиву для більшої наглядності а саме N = 1000000000;

Порівнюватимемо такі масиви даних:
-Рандомні числа

-Відсортований масив

-Розвернутий масив, тобто(N, N-1, ... , 3, 2, 1)

-Масив із однаковими числами

-Pipe Organ, тобто(1, 2, 3... N, N-1, N-2... 1)

Також було реалізовано тестування результату, що перевіряє чи взагальному правильно відпрацьовує qsort(), вкінці отримуємо інформативний вивід в консоль, по якому легко можна зробити висновки, які типи масивів, сортуються швидше, а які повільніше, тобто у виводі ми побачимо кількість порівнянь, та час за який було виконано сортування, останній параметр на вивід це Correct/Fail, тобто результат проходження нашого тестування.

Відповідь:
Експеримент демонструє, що без прапорця O_APPEND зв'язка lseek(SEEK_END) + write() не є атомарною, через що між визначенням кінця файлу та самим записом інший процес встигає втрутитися, призводячи до стану гонки (race condition), затирання даних та некоректного фінального розміру файлу (1801 байт замість 1800). Натомість використання O_APPEND змушує ядро операційної системи виконувати перехід у кінець файлу та запис як єдину неподільну операцію, що гарантує цілісність структури файлу, ідеальний збіг очікуваного розміру та збереження кожного записаного байта навіть при інтенсивному паралельному доступі.
